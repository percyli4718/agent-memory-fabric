# Agent Memory Fabric 策略记录

## 项目定位

项目定位为一个面向多种 agent 工具平台的开源记忆基础设施，而不是某一家模型厂商或某一种 IDE 的插件。

建议定位语句：

- 项目名：`Agent Memory Fabric`
- slogan：`One memory layer for every agent.`

核心价值主张：

- 跨会话搜索与复用记忆
- 跨仓库、跨项目、跨工具共享记忆
- 以协议、接口、运行时、治理为开源核心
- 企业定制部分通过扩展实现，而不是污染开源核心

## 我们达成的一致

1. 真正适合做跨会话持久化记忆的核心是 `MCP`
2. `skill` 更适合封装工作流，而不是直接承担存储职责
3. `plugin` 更适合打包和分发 `skill + MCP + 示例配置`
4. `agent` 或 `subagent` 适合执行读写记忆流程，不适合作为记忆本体
5. 企业级小团队不应该依赖单机插件式本地记忆作为唯一方案
6. 应该采用开源核心加企业扩展的边界划分

## 建议的产品边界

开源核心负责：

- 协议
- 接口
- 运行时
- schema
- policy
- 审计能力
- 参考部署方式

企业扩展负责：

- 组织级身份与权限
- 私有数据连接器
- 多租户运营能力
- 供应链和审计集成
- 更深的合规策略

## 技术架构建议

推荐拆成四层：

1. `Memory Core`
   负责记忆模型、检索、压缩、去重、作用域和保留策略
2. `MCP Server`
   负责暴露标准工具接口，供不同 agent 调用
3. `Client Adapters`
   负责适配不同平台，例如 Codex、Claude Code、Cursor、Gemini 类工具，以及国内平台
4. `Policy and Prompt Packs`
   负责以 `AGENTS.md`、模板和仓库规则约束读写记忆的行为

## 安全和治理边界

默认约束建议：

- 不存储原始密钥、令牌、密码
- 不默认保存完整原始会话
- 优先存结构化摘要和可审计元数据
- 支持租户、项目、仓库、用户四级作用域
- 支持删除、修正、脱敏、保留期
- 支持写入和读取权限分离

## 开源项目应达到的标准

建议最少包括：

- `README.md`
- `AGENTS.md`
- `SECURITY.md`
- `CONTRIBUTING.md`
- `CODE_OF_CONDUCT.md`
- `LICENSE`
- SBOM
- 发布签名
- 供应链来源证明
- 漏洞披露流程

## v0.1 目标

第一阶段先做小而完整的闭环：

- 核心 memory schema
- 基础 MCP server
- SQLite 与 PostgreSQL 参考存储
- 项目级与仓库级记忆
- 会话总结写入
- 基础搜索能力
- 本地部署示例

## 整体规划补充

为了避免后续换模型、换会话、换协作者时上下文断裂，项目需要有明确的总路线图与阶段出口标准。

建议整体阶段如下：

1. `Phase 0`
   立项、边界、命名、架构和开源基线
2. `Phase 1`
   v0.1 开源核心，验证 MCP 跨会话记忆闭环
3. `Phase 2`
   v0.2 团队可用，补齐审计、保留、删除、脱敏和基础部署
4. `Phase 3`
   v0.3 企业可用，补齐多租户、安全供应链、兼容矩阵和合规说明
5. `Phase 4`
   v1.0 生态稳定，形成稳定 schema、MCP 契约和适配层

配套文档建议包括：

- `docs/roadmap.md`
- `docs/model-handoff.md`
- `docs/architecture.md`
- `docs/threat-model.md`

其中 `docs/model-handoff.md` 是为“大模型额度用完后切换到下一个模型继续做”准备的关键文档。

## 设计原则

1. 协议中立
2. 模型中立
3. 存储可替换
4. 安全默认开启
5. 作用域和审计优先
6. 先做稳定可治理的记忆层，再做高级智能化能力
